# 共通ポリシー

autopilot ワークフロー全体で適用される共通ルール。各サブエージェントはこのポリシーに従うこと。

---

## コーディングポリシー

速さより丁寧さ、実装の楽さよりコードの正確さを優先する。

| 原則           | 基準                               |
| -------------- | ---------------------------------- |
| Simple > Easy  | 書きやすさより読みやすさを優先     |
| DRY            | 3回重複したら抽出                  |
| コメント       | Why のみ。What/How は書かない      |
| 関数サイズ     | 1関数1責務。30行目安               |
| ファイルサイズ | 目安として300行                    |
| Fail Fast      | エラーは早期に検出。握りつぶさない |

**フォールバック・デフォルト引数の禁止:** 値の流れを不明瞭にするコードは書かない。必須データへのフォールバック（`user?.id ?? 'unknown'`）、デフォルト引数の濫用、try-catch で空値返却は禁止。

**禁止事項:**

- フォールバック原則禁止
- 説明コメント（What/How のコメント）
- 未使用コード
- any 型
- オブジェクト/配列の直接変更
- console.log / デバッグコード残留
- 各所での try-catch（エラーの握りつぶし）
- 後方互換・Legacy 対応の自発的追加

---

## テストポリシー

全ての振る舞いの変更には対応するテストが必要。全てのバグ修正にはリグレッションテストが必要。

| 原則            | 基準                               |
| --------------- | ---------------------------------- |
| Given-When-Then | テストは3段階で構造化              |
| 1テスト1概念    | 複数の関心事を混ぜない             |
| 振る舞いを検証  | 実装の詳細ではなく振る舞いをテスト |
| 独立性          | 他のテストや実行順序に依存しない   |
| 再現性          | 毎回同じ結果                       |

---

## レビューポリシー

| 原則             | 基準                                   |
| ---------------- | -------------------------------------- |
| 即座修正         | 軽微でも「次のタスク」にしない         |
| 曖昧さ排除       | ファイル・行・修正案を具体的に示す     |
| ファクトチェック | 推測ではなく実コードを確認してから指摘 |

**スコープ判定:**

- 今回の変更で導入された問題 → ブロッキング（REJECT / REQUEST CHANGES）
- 変更ファイル内の既存問題 → ブロッキング（ボーイスカウトルール）
- 変更外ファイルの問題 → 非ブロッキング（記録のみ）

**REJECT 基準:** テストがない新振る舞い、any 型、フォールバック値の乱用、説明コメント、未使用コード、エラーの握りつぶし、TODO 放置、3箇所以上の重複、特定実装の汎用層への漏洩。

---

## ファクトチェックポリシー

分析・設計・実装で使用する情報は必ずソース・オブ・トゥルースで裏取りする。推測で判断しない。

| 確認対象         | ソース・オブ・トゥルース |
| ---------------- | ------------------------ |
| コードの振る舞い | 実際のソースコード       |
| 設定値           | 実際の設定ファイル       |
| API              | 実際の実装コード         |
| 型               | 型定義ファイル           |
| 依存関係         | パッケージマニフェスト   |

---

## AI 生成コードの注意事項

AI コーディングアシスタントが陥りやすいアンチパターンを自覚し、意識的に避けること。

**禁止パターン:**

- 不確実なときにフォールバックで隠す → 禁止。エラーを明示的に処理する
- 「念のため」で未使用コードを書く → 禁止。必要になったときに追加する
- 設計判断を勝手にする → 報告して判断を仰ぐ
- 後方互換・Legacy 対応を勝手に追加する → 絶対禁止
- 存在しない API/メソッドを呼び出す → 実コードで確認してから使う
- 要求されていない追加機能を実装する → スコープ内に留まる

**検出基準:**

| カテゴリ                     | 確認内容                                                     |
| ---------------------------- | ------------------------------------------------------------ |
| 仮定の検証                   | 実装は要求と一致しているか、既存パターンに合っているか       |
| もっともらしいが間違っている | 幻覚 API、古いパターン、過剰/過小エンジニアリング、配線忘れ  |
| スコープクリープ             | 追加機能、早すぎる抽象化、過剰設定、不要 Legacy 対応         |
| デッドコード                 | 未使用関数、到達不能コード、孤立エクスポート                 |
| フォールバック濫用           | 必須データへのフォールバック、デフォルト引数の全呼び出し省略 |
| 後方互換                     | deprecated + 使用箇所なし、新旧 API 共存、移行済みラッパー   |
